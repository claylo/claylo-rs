# claylo-rs: Rust CLI Project Template

> A Copier template that generates production-ready Rust CLI workspaces with modular feature selection.

Source: `gh:claylo/claylo-rs`

## What This Template Provides

When an AI agent is helping plan, brainstorm, or implement features in a project generated by claylo-rs, it should understand that the following infrastructure **already exists** and should not be recreated.

## Project Generation

```bash
# Create a new project (non-interactive)
claylo-rs new ./my-tool --preset standard -y

# With feature flags (+ enables, - disables)
claylo-rs new ./my-tool --preset standard +otel+mcp-bench -y

# Supply identity fields for fully unattended generation
claylo-rs new ./my-tool --preset standard --owner myorg --copyright "My Name" -y

# Update an existing project
claylo-rs update
```

The `-y` / `--yes` flag accepts preset defaults without prompting. Without it, copier will prompt for any values not supplied via flags or defaults files.

### Defaults File

If a `.claylo-rs.defaults.yaml` file exists in the current directory or any parent directory up to `/`, it is automatically loaded as default values. This lets teams or individuals pre-fill identity fields (owner, copyright, description) so `-y` works without extra flags:

```yaml
# ~/.claylo-rs.defaults.yaml — personal defaults
owner: myorg
copyright_name: "My Name"
project_description: "A Rust CLI tool"
```

For full option details, run `claylo-rs --help`.

### Presets

Four presets configure coherent defaults:

| Preset | What you get |
|--------|-------------|
| `minimal` | CLI binary only. No config, no logging, no telemetry. |
| `library` | Core library + benchmarks + release automation. No CLI binary or binary distribution. |
| `standard` | CLI + core library + config + JSONL logging + documentation site + release automation |
| `full` | Everything in standard + benchmarks + OpenTelemetry + community files + editor configs + env files |

## Workspace Structure

Generated projects use a Cargo workspace with crates under `crates/`:

```
my-tool/
├── crates/
│   ├── my-tool/              # CLI binary (clap, anyhow)
│   └── my-tool-core/         # Library (thiserror, domain logic)   [+core]
├── xtask/                    # Build automation (completions, man pages)
├── config/                   # Example config files (TOML, YAML)   [+config]
├── site/                     # Astro Starlight documentation site   [+site]
├── npm/                      # NPM binary distribution packages    [+cli +releases]
├── bench-reports/            # Benchmark output storage             [+bench]
├── dist/share/               # Generated completions and man pages
├── .github/                  # CI/CD workflows, actions, templates  [+github]
├── .claude/                  # Agent config, skills, commands       [+claude]
├── scripts/add-crate         # Workspace crate scaffolding tool
├── Cargo.toml                # Workspace manifest
├── deny.toml                 # Security/license policy
├── rust-toolchain.toml       # Pinned toolchain
├── cliff.toml                # git-cliff changelog config           [+releases]
└── .justfile                 # Task runner recipes
```

**Architecture principle: thin CLI, fat core.** The CLI crate handles argument parsing and I/O. Feature logic, types, and dependencies belong in the `-core` library crate. This keeps the core testable without subprocess gymnastics.

## What's Already Built — Don't Reinvent These

### CLI Framework

- **clap** (derive) for argument parsing with subcommands
- Color output control (`--color auto|always|never`)
- `--version`, `--version-only` flags
- `--chdir` for directory change before execution
- Exit codes following standard conventions

### Built-in Subcommands

| Command | Included | What it does |
|---------|----------|-------------|
| `info` | Always | Version, build info, config status. Supports `--json`. |
| `doctor` | When `+config` | Diagnostic output: config paths, XDG dirs, env vars. Supports `--json`. |
| `serve` | When `+mcp` | MCP server over stdio transport. |

### Adding New Subcommands

1. Create `crates/<project>/src/commands/your_cmd.rs`
2. Add variant to `Commands` enum in `crates/<project>/src/lib.rs`
3. Wire it up in `match cli.command` in `main.rs`
4. Add integration tests in `crates/<project>/tests/`

There is a Claude Code skill at `.claude/skills/adding-commands/` that guides this process.

### Configuration System (when `+config`)

- **figment**-based configuration loading
- TOML and YAML format support
- Hierarchical discovery: XDG config dirs → project directory → parent directories
- `ConfigLoader` with builder pattern
- `ConfigError` / `ConfigResult` types via thiserror
- Example config files in `config/` directory

**Do not build a config system.** Use `ConfigLoader::new().with_project_search(&cwd)`.

### Observability (when `+jsonl` or `+otel`)

- **tracing** + **tracing-subscriber** for structured logging
- **tracing-appender** for daily-rotated JSONL file logging
- `RUST_LOG` environment variable support
- `ObservabilityConfig` struct for centralized setup
- OpenTelemetry OTLP export (when `+otel`)
  - Reads `OTEL_EXPORTER_OTLP_ENDPOINT` from environment
  - Merges with config file `otel_endpoint` setting
  - Adds **tokio** async runtime automatically

**Do not add logging infrastructure.** It exists. Use `tracing::info!()` etc.

### Error Handling

- **thiserror** in the `-core` library crate for typed errors
- **anyhow** in the CLI binary for ad-hoc error propagation
- Error types defined in `crates/<project>-core/src/error.rs`

**Do not add `Box<dyn Error>` or invent error types.** Extend the existing `thiserror` enum.

### Interactive CLI (opt-in)

- **inquire** (when `+inquire`) — Text, Select, Confirm prompts
- **indicatif** (when `+indicatif`) — progress bars and spinners

### MCP Server (when `+mcp`)

- Model Context Protocol server scaffold
- stdio transport with tokio async runtime
- Resource discovery and tool invocation handlers
- Logging routed to JSONL files (not stdout — stdout is the MCP transport)

## Task Runner (justfile)

All development tasks have `just` recipes. **Never run raw cargo commands when a recipe exists.**

### Core Recipes (always present)

| Recipe | What it does |
|--------|-------------|
| `just bootstrap` | First-time setup: MSRV check, hook install, build, generate completions/man pages |
| `just check` | Full quality gate: fmt + clippy + deny + test + doc-test |
| `just test` | `cargo nextest run` (NOT `cargo test`) |
| `just clippy` | Lint with pinned toolchain, all targets, all features |
| `just fmt` | `cargo fmt --all` |
| `just fix` | Auto-fix clippy warnings |
| `just deny` | Security and license audit |
| `just cov` | Coverage via `cargo llvm-cov nextest` |
| `just watch` | Watch mode for tests |
| `just watch-clippy` | Watch mode for clippy |

### Dependency Management (always present)

| Recipe | What it does |
|--------|-------------|
| `just outdated` | Check for outdated deps (root only) |
| `just update` | Safe update: respects semver, only touches Cargo.lock |
| `just upgrade` | Upgrade Cargo.toml to latest compatible versions |
| `just upgrade-breaking` | Upgrade to latest incompatible versions |
| `just refresh` | Update + test + clippy |
| `just monthly` | Full maintenance: upgrade + test + clippy + release build |

### Documentation Site (when `+site`)

| Recipe | What it does |
|--------|-------------|
| `just site-install` | Install site dependencies |
| `just site-dev` | Start dev server |
| `just site-build` | Production build |
| `just site-preview` | Preview production build |

### Benchmarks (when `+bench`)

| Recipe | What it does |
|--------|-------------|
| `just bench` | Run all benchmarks via xtask |
| `just bench-divan` | Wall-clock microbenchmarks only |
| `just bench-cli` | CLI benchmarks via hyperfine |
| `just bench-gungraun` | Instruction-count benchmarks (when `+gungraun`) |

### Release Management (when `+releases`)

| Recipe | What it does |
|--------|-------------|
| `just changelog` | Generate CHANGELOG.md via git-cliff |
| `just changelog-preview` | Preview unreleased changes |
| `just bump` | Auto-calculate and apply next semantic version |
| `just release v1.2.3` | Full release with pre-checks, changelog, tag |
| `just release-check` | Pre-release validation (clean tree, tests, clippy, deny, release build) |

### Markdown Linting (when `+md`)

| Recipe | What it does |
|--------|-------------|
| `just mdlint` | Lint markdown (rumdl or markdownlint) |
| `just mdfix` | Auto-fix markdown issues |

### Workspace Scaffolding (always present)

| Recipe | What it does |
|--------|-------------|
| `just add-crate lib my-utils` | Add a library crate |
| `just add-crate bin my-daemon` | Add a binary crate |
| `just add-crate internal my-macros --derive` | Add proc-macro crate |

The `add-crate` script generates `Cargo.toml` with workspace inheritance, conventional dependencies, and runs `cargo check` to verify.

## CI/CD (when `+github`)

### GitHub Actions Workflows

| Workflow | Trigger | What it does |
|----------|---------|-------------|
| `ci.yml` | PR, push to main | lint (fmt + clippy) → test (nextest + doctests) → cargo-deny → MSRV check |
| `lint-pr.yml` | PR title | Validates conventional commit format in PR titles |
| `dependabot-issues.yml` | Dependabot PR | Auto-creates tracking issues for dependency updates |
| `deploy-site.yml` | Push to main | Build and deploy Starlight docs (when `+site`) |
| `benchmarks.yml` | Manual/scheduled | Run benchmarks and store results (when `+bench`) |
| `release.yml` | Push to main | Auto-release via git-cliff (when `+releases`) |
| `cd.yml` | Tag push `v*.*.*` | Cross-platform builds, GitHub Release, optional publishing |

### CD Build Matrix (when `+releases`)

Cross-compilation for 8 targets:

| Platform | Targets |
|----------|---------|
| Linux | x86_64-gnu, x86_64-musl, aarch64-gnu, aarch64-musl |
| macOS | x86_64-apple-darwin, aarch64-apple-darwin |
| Windows | x86_64-pc-windows-msvc, aarch64-pc-windows-msvc |

### Optional Publishing

Publishing targets are controlled by **GitHub repo variables**, not template flags — they can be toggled after project generation without re-running the template:

| Variable | What it enables |
|----------|----------------|
| `CRATES_IO_ENABLED` | Publish to crates.io |
| `HOMEBREW_ENABLED` | Update Homebrew formula |
| `NPM_ENABLED` | Publish to npm (platform-specific binary packages) |
| `DEB_ENABLED` | Build .deb packages |
| `RPM_ENABLED` | Build .rpm packages |
| `SBOM_ENABLED` | Generate CycloneDX SBOM |
| `GPG_SIGNING_ENABLED` | Sign release artifacts |

## Documentation Site (when `+site`)

- **Astro** + **Starlight** theme
- Content in `site/src/content/docs/` (MDX)
- Guides and CLI reference pages pre-generated
- LLM-friendly endpoints: `/llms.txt` and `/llms-full.txt`
- Deployment: GitHub Pages (default), Cloudflare Pages (via GH Actions or git integration)
- Package manager: npm (default), pnpm, bun, or yarn

## Claude Code Integration (when `+claude`)

Generates `.claude/settings.json` with pre-configured permissions for common development tools (cargo, git, just, gh) and security-conscious deny rules (no force-push, no rm -rf, no reading secrets).

## Feature Flags

All flags toggle with `+flag` / `-flag` on the `claylo-rs` command line.

### Structure & Runtime

| Flag | Description |
|------|-------------|
| `+cli` | CLI binary crate (default: on) |
| `+core` | Separate `-core` library crate |
| `+config` | Configuration file support (figment, TOML/YAML) |
| `+jsonl` | Structured JSONL file logging (daily-rotated) |
| `+otel` | OpenTelemetry trace export (adds tokio) |
| `+mcp` | MCP server as `serve` subcommand (adds tokio) |
| `+inquire` | Interactive prompts (Text, Select, Confirm) |
| `+indicatif` | Progress bars and spinners |

### Performance

## Post-Generation Setup

After `claylo-rs new`, run `just bootstrap`. It handles MSRV validation, git hooks, formatting, building, shell completions, man pages, site deps, and repo settings — everything needed to go from template output to working project.

## Architecture Decision Records

Generated projects include `docs/decisions/` for MADR-format ADRs. The `capturing-decisions` Claude Code skill provides a template.

## Updating Projects

```bash
claylo-rs update                  # update from latest template (inline conflict markers by default)
claylo-rs update --conflict rej   # use .rej sidecar files instead
```
